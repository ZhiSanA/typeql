# 类型查询引擎（TypeQL）设计修改提案

## 1. Where 多级关联查询

### 1.1 现状分析

当前 `where` 筛选系统只支持**当前实体自身列**的筛选条件：

- `common.ts` 的 `buildTableTypes()` 仅为 `meta.ownColumns` 生成 filter fields
- `resolvers.ts` 的 `convertFilters()` 仅在 `ownColumns` 中查找列，忽略所有 relation
- `filterInput` 生成后，嵌套字段只包含 `or` 组合，没有 relation 嵌套筛选入口

对于关联实体的筛选（如 `where: { author: { name: { eq: "test" } } }`），完全无法工作。

### 1.2 目标

支持 GraphQL 查询中通过关联关系进行多级嵌套筛选：

```graphql
query {
  articles(where: { author: { name: { eq: "Alice" } } }) {
    id
    title
  }
}
```

```graphql
query {
  articles(where: { author: { profile: { bio: { like: "engineer" } } } }) {
    id
    title
    content
  }
}
```

支持所有关系类型：

- **ManyToOne**（articles → author，通过外键聚合）
- **OneToOne**（双向）
- **OneToMany**（逆方向）
- **ManyToMany**（通过 junction table）

### 1.3 设计方案

#### 1.3.1 Filter 类型生成扩展（`common.ts`）

在 `buildTableTypes()` 中，为每个实体的 `relationMap` 条目生成嵌套 filter：

```
ArticleFilter {
  id: IntFilter
  title: StringFilter
  author: Article_author_Filter    # ← 新增：关联筛选入口
  or: [Article_OrFilter]
}

Article_author_Filter {
  id: IntFilter
  name: StringFilter
  profile: Article_author_profile_Filter  # ← 支持多级嵌套
  or: [Article_author_OrFilter]
}
```

**策略：** 对 filter 输入类型生成使用**实际深度限制**（默认 2 级，可通过配置调整），且引用链不做循环引用（A → B → A 不继续展开），在递归时传入 `visitedEntities: Set<string>` 防止无限递归。

**与 output type 的区别：**

- output type 通过 GraphQL `fields: () => ({})` thunk 延迟求值来处理循环引用
- filter type 不能这样，因为 `GraphQLInputObjectType` 的 fields 会在构建时立即求值用于序列化
- 因此 filter 必须使用**短路策略**：遇到已访问的实体名时，只生成自身列的 filter，不展开 relation

#### 1.3.2 Where 解析扩展（`resolvers.ts`）

关系筛选使用 TypeORM 的 `relations` 结合嵌套 `where` 实现：

```typescript
// 转化前
{
  author: {
    name: {
      eq: 'Alice';
    }
  }
}

// 转化后（TypeORM 格式）
{
  author: {
    name: 'Alice';
  } // TypeORM 原生支持嵌套 relation where
}
```

核心改动：

1. 在 `resolveWhere()` 中，当遇到非 ownColumns 的 key 时，将其识别为 relation 路径
2. 递归处理嵌套结构，对叶子节点应用 `convertFilters()` 的 operator 映射
3. 在 `repo.find()` 时需要补充 `relations` 字段（或者让 TypeORM 自动处理）

实际上 TypeORM 的 `find()` 支持嵌套的 `where` + `relations`，我们需要：

- 从 where 结构中提取所有 relation 路径
- 展开为 `relations: ['author', 'author.profile']` 格式
- 并将嵌套 where 原样传递给 TypeORM

**函数设计：**

```typescript
// resolvers.ts

function resolveRelations(
  argsWhere: any,
  meta: EntityMetadata,
  relationMap: RelationMap,
): string[] {
  const relations: string[] = [];
  if (!argsWhere) return relations;

  for (const key of Object.keys(argsWhere)) {
    if (key === 'or' || key === 'OR') continue;
    const relInfo = relationMap[meta.targetName]?.[key];
    if (relInfo) {
      relations.push(key);
      const subRelations = resolveRelations(
        argsWhere[key],
        targetMeta,
        relationMap,
      );
      relations.push(...subRelations.map((r) => `${key}.${r}`));
    }
  }
  return relations;
}
```

`resolveWhere()` 返回结构从 `FindOptionsWhere` 调整为 `{ where: FindOptionsWhere, relations: string[] }`。

## 2. 关联字段参数化（Relation Field Arguments）

### 2.1 问题描述

当前列表类型关联字段（OneToMany/ManyToMany）在 GraphQL 中是无参字段：

```graphql
query {
  articles {
    comments {
      # ← 无法传参，一次性加载所有关联数据
      id
      title
    }
  }
}
```

当关联数据量很大时（如一篇文章有数万条评论），会导致：

- 单次查询加载过多数据，内存暴涨
- 数据库查询性能差
- 网络传输体积过大

Helium 目前的实现也缺乏这种限制机制。

### 2.2 目标

让列表关联字段支持可选的参数化查询：

```graphql
query {
  articles {
    comments(
      where: { status: { eq: "published" } }
      orderBy: { createdAt: { direction: DESC, priority: 1 } }
      limit: 20
      offset: 0
    ) {
      id
      title
      createdAt
    }
  }
}
```

- 对 `isOne = true`（ManyToOne/OneToOne）的关系：保持无参（单条数据不需要分页）
- 对 `isOne = false`（OneToMany/ManyToMany）的关系：增加可选的 `where`、`orderBy`、`limit`、`offset` 参数
- 所有参数均为可选项，不传时行为与现有逻辑一致

### 2.3 设计方案

#### 2.3.1 类型构建扩展（`common.ts`）

在 `buildOrGetType()` 的 fields 构造中，对列表类型关系字段添加 args：

```typescript
for (const [relName, relInfo] of Object.entries(relations)) {
  const targetMeta = entityMap[info.targetEntityName];
  const targetNames = resolveNames(info.targetEntityName, typeNameMapper);
  const targetType = buildOrGetType(targetNames.typeName, ...);

  if (info.isOne) {
    // 单条关联：无参
    fields[relName] = { type: targetType, resolve: fieldResolver };
  } else {
    // 列表关联：支持参数
    const targetFilter = filterCache.get(targetMeta.targetName);
    const targetOrder = orderCache.get(targetMeta.targetName);
    fields[relName] = {
      type: new GraphQLList(new GraphQLNonNull(targetType)),
      args: buildRelationArgs(targetFilter, targetOrder),
      resolve: fieldResolver,
    };
  }
}
```

但是这里有一个问题：`buildOrGetType` 目前不接收 filterInput/orderInput。需要将类型构建流程调整为：

1. 先生成所有的 filter/order 类型（已完成）
2. 再生成 output 类型时，传入已构建好的 filter/order 映射

或者，将 filter/order 的生成提前到 `generateTypes()` 中，并在 `buildOrGetType` 中通过共享 Map 获取。

**实际做法：** filter/order 已经在 `generateTypes()` 中生成了，但它们存储在 `typeOutputs.filters`/`typeOutputs.orders` 中。需要在 `buildOrGetType()` 中也能访问到这些，或者提前将 `filterCache`/`orderCache` 作为全局/模块级 Map。

方案：在 `common.ts` 中添加模块级 `relArgsFilterCache` 和 `relArgsOrderCache`，由 `generateTypes()` 在生成时填充，`buildOrGetType()` 在构造 fields 时读取。

#### 2.3.2 解析器扩展（`resolvers.ts`）

当前的关系解析器 `createRelationResolver()` 返回的函数签名是 `(source, _args, context) => Promise<any>`，其中的 `_args` 目前未被使用。

需要让解析器接收并处理这些额外参数：

```typescript
function createRelationResolver(
  ds,
  meta,
  relInfo,
  relationMap,
): (source, args, context) => Promise<any> {
  // ... 现有逻辑 ...

  return async (source: any, args: any, context: any) => {
    // 如果 source 上已有预加载的数据，直接返回（对 GraphsQL 字段解析器的 dataLoader 模式）
    if (source[propertyName] !== undefined && !hasRelationArgs(args)) {
      return source[propertyName];
    }

    // 解析 where/orderBy/limit/offset
    const where = args?.where
      ? resolveWhere(args.where, targetMeta, relationMap)
      : undefined;
    const order = args?.orderBy ? convertOrderBy(args.orderBy) : undefined;
    const take = args?.limit ?? undefined;
    const skip = args?.offset ?? undefined;

    if (isManyToMany) {
      // ManyToMany：使用 QueryBuilder 支持参数
      return queryManyToMany(source, relInfo, { where, order, take, skip });
    }

    if (relInfo.isOwning) {
      // ManyToOne（isOne=true）：不需要分页，保持现有逻辑
      return loadOwningSide(source, relInfo);
    } else {
      // OneToMany：带参查询
      return queryOneToMany(source, relInfo, { where, order, take, skip });
    }
  };
}
```

**关于 Batch Loader：** 当带有 where/order/limit/offset 参数时，批处理不再适用（每个请求的过滤条件不同），应退化为直接查询。无参时保持现有 batch loader 逻辑。

#### 2.3.3 批量查询的退化策略

| 场景                           | 策略                       |
| ------------------------------ | -------------------------- |
| 无参列表查询                   | 保持 BatchLoader 批处理    |
| 带参列表查询                   | 退化为直接查询，不再批处理 |
| 单条查询（ManyToOne/OneToOne） | 保持 BatchLoader，始终无参 |

#### 2.3.4 安全性考虑

- **默认值：** `limit` 不传时不做限制，保持向后兼容
- **最大限制：** 可通过配置 `maxRelationLimit` 进行上限控制（可选）

## 3. Delete 返回值修复

### 3.1 问题描述

当前 `makeDelete()` 实现：

```typescript
function makeDelete(ds, meta, fi, lt, cols) {
  return {
    type: lt, // GraphQLNonNull(GraphQLList(GraphQLNonNull(selectType)))
    resolve: async (_s, args) => {
      const entities = await repo.find({
        where: resolveWhere(args['where'], cols!),
      });
      if (!entities.length) return [];
      return remapToGraphQLArrayOutput(await repo.remove(entities));
    },
  };
}
```

`repo.remove()` 返回的实体中，被删除后的数据字段多数为 `undefined`/`null`，而 `remapToGraphQLArrayOutput` 会 `delete` 这些字段。结果返回的对象缺少非空字段，导致 GraphQL 非空校验失败：

```
Cannot return null for non-nullable field Xxx.id
```

### 3.2 解决方案

#### 方案 A：修改返回类型为 DeleteResult（推荐）

将 delete 的返回类型从实体列表改为删除结果对象，参考 Helium 的 `DeleteEntitiesResult`：

```graphql
type DeleteResult {
  affected: Int
  raw: [String]
}
```

**改动点：**

1. `common.ts`：新增公共返回类型 `DeleteResult`，包含 `affected` 和 `raw` 字段
2. `resolvers.ts`：`makeDelete` 使用 `repo.delete()` 替代 `find + remove`，返回 `{ affected, raw }`

```typescript
function makeDelete(ds, meta, fi, lt, cols) {
  return {
    type: deleteResultType, // GraphQLObjectType { affected, raw }
    args: fi ? { where: { type: fi } } : {},
    resolve: async (_s, args) => {
      const repo = ds.getRepository(target);
      const where = resolveWhere(args['where'], meta, relationMap);
      const result = await repo.delete({ where: where.where });
      return { affected: result.affected ?? 0, raw: result.raw ?? [] };
    },
  };
}
```

## 6. 完整使用示例

```graphql
# 带关联筛选的主查询 + 关联字段分页
query {
  articles(where: { author: { name: { eq: "Alice" } } }, limit: 10) {
    id
    title
    author {
      id
      name
    }
    comments(
      where: { status: { eq: "approved" } }
      orderBy: { createdAt: { direction: DESC, priority: 1 } }
      limit: 5
    ) {
      id
      content
      createdAt
    }
  }
}
```

```graphql
# 删除操作
mutation {
  deleteArticle(where: { status: { eq: "draft" } }) {
    affected
  }
}
```
